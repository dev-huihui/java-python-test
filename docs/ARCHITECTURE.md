# 프로젝트 설명 — Java ↔ Python 검증 연동

이 문서는 이 프로젝트가 **무엇을, 왜, 어떻게** 하는지와,
지금 쓰고 있는 **두 가지 실행 방식(프론트/백엔드)** 및 그 **주의점·단점**을 설명합니다.

## 1. 핵심 아이디어

> **검증 규칙(로직)은 Python 파일에만 두고, Java 프로젝트는 그 Python 을 "실행하는 통로" 역할만 한다.**

- 검증 로직: `src/main/resources/python/` 의 `validation.py`(회원가입), `validators.py`(다양한 검증기)
- 실행 위치: **프론트엔드(브라우저)** 또는 **백엔드(서버)** 중 선택
- 같은 `.py` 파일을 두 위치에서 공용으로 사용하므로, 규칙을 바꾸면 한 곳(Python)만 수정하면 됩니다.

```
                ┌─────────────────────────────┐
                │   검증 규칙 (Python 파일)     │
                │  validation.py / validators.py │
                └───────────────┬─────────────┘
              (같은 파일을 두 경로가 공용)
          ┌───────────────────┴───────────────────┐
          ▼                                         ▼
 ① 프론트엔드 (브라우저)                    ② 백엔드 (서버, JVM)
   Pyodide(WASM)로 .py 실행                 ProcessBuilder로 python 실행
```

## 2. 방식 ① — 프론트엔드 실행 (Pyodide)

**동작:** 브라우저가 [Pyodide](https://pyodide.org)(CPython을 WebAssembly로 포팅한 런타임)를 로드하고,
서버가 내려준 `.py` 텍스트(`/python/{name}`)를 그 자리에서 실행합니다. 서버 왕복 없이 브라우저 안에서 검증이 끝납니다.

**장점**
- 서버에 Python 설치가 필요 없다.
- 네트워크 왕복이 없어 입력 즉시 피드백(빠른 UX)이 가능하다.
- 서버 부하가 없다.

**주의점 · 단점**
- **보안 검증으로 쓰면 안 된다.** 브라우저에서 도는 코드는 사용자가 얼마든지 우회·조작할 수 있다. 어디까지나 *편의(UX)용 사전 검증*이며, **실제 신뢰가 필요한 검증은 반드시 서버에서 다시 해야 한다.**
- **검증 로직이 클라이언트에 그대로 노출**된다. `validators.py` 소스가 브라우저로 전송되므로, 민감한 규칙·비밀 값을 넣으면 안 된다.
- **초기 로딩이 무겁다.** Pyodide 런타임은 수 MB 이상이라 첫 실행 시 다운로드 지연이 있고, **CDN 의존**(현재 jsDelivr)이라 인터넷이 없으면 동작하지 않는다. (오프라인이 필요하면 Pyodide를 로컬 정적 리소스로 번들해야 함)
- Pyodide는 순수 Python + 일부 패키지만 지원한다. C 확장에 의존하는 라이브러리는 제약이 있을 수 있다(다수는 지원되지만 전부는 아님).
- 브라우저 탭에 Python 런타임이 상주하므로 메모리 사용이 늘어난다.

## 3. 방식 ② — 백엔드 실행 (ProcessBuilder)

**동작:** 스프링 컨트롤러가 요청을 받으면, `PythonValidationService`가 classpath의 `.py`를 임시 폴더로 꺼내고
`ProcessBuilder`로 `python <script>`를 실행한다. 입력은 **stdin(JSON)** 으로 넘기고 결과는 **stdout(JSON)** 으로 받는다.

**장점**
- 서버에서 실행되므로 **신뢰할 수 있는 검증**(위·변조 불가)이다.
- 검증 로직이 클라이언트에 노출되지 않는다.
- Java에서 언어에 독립적으로 Python 자산을 재사용할 수 있다(연동 방식이 단순).

**주의점 · 단점**
- **매 요청마다 Python 프로세스를 새로 생성**한다. 프로세스 fork 비용(수십~수백 ms)이 있어 **고빈도 요청에는 비효율적**이다. 개선하려면 장기 실행 파이썬 프로세스/서버(Py4J·gRPC·REST 마이크로서비스), 배치 처리, 또는 JVM 내 실행(GraalPy) 등을 고려한다.
- **서버 환경에 Python 설치·버전·경로 의존**이 생긴다. 배포 대상마다 Python이 있어야 하고 `app.python.command`를 맞춰야 한다.
- **인코딩 함정.** Windows에서 파이프의 기본 인코딩이 시스템 로케일(예: CP949)이라 한글이 깨질 수 있다. 이 프로젝트는 `cli.py`/`field_cli.py`에서 stdin/stdout을 UTF-8로 고정하고 `PYTHONIOENCODING=utf-8`을 지정해 해결했다.
- **보안.** 사용자 입력은 stdin(JSON)으로만 전달하고 명령행 인자/셸로 넘기지 않아 명령어 주입을 피했다. 임의 스크립트/경로를 실행하지 않도록 실행 대상은 화이트리스트로 고정해야 한다.
- **동시성.** 요청당 1 프로세스라 동시 요청이 많으면 프로세스 수가 폭증할 수 있다. 실제 서비스에서는 실행 수 제한(세마포어/스레드풀)이 필요하다.
- **타임아웃·스트림 처리.** 무한 루프/행을 막기 위해 타임아웃(현재 10초)을 둔다. 또한 출력이 매우 클 경우 stdout/stderr 버퍼가 차면서 교착(deadlock)이 생길 수 있으니, 큰 출력이 예상되면 스트림을 별도 스레드로 읽거나 리다이렉트해야 한다(현재는 출력이 작아 문제없음).

## 4. 언제 무엇을 쓰나

| 상황 | 권장 방식 |
|------|-----------|
| 입력 즉시 피드백 등 UX 편의 | 프론트(Pyodide) |
| 실제로 신뢰해야 하는 검증(저장·인증·결제 등) | **백엔드(서버)** |
| 오프라인/인터넷 불가 환경 | 백엔드 또는 Pyodide 로컬 번들 |
| 초당 많은 요청 처리 | 백엔드지만 프로세스 재사용/전용 서비스로 개선 |

**원칙:** 프론트 검증은 어디까지나 보조 수단이고, **최종 신뢰 경계는 서버**다. 두 방식을 함께 쓸 때도 서버 검증을 생략하면 안 된다.

## 5. 참고 — 다른 Java↔Python 연동 방식

이 프로젝트는 가장 단순한 **ProcessBuilder(외부 프로세스 호출)** 방식을 쓴다. 대안:

- **GraalPy (GraalVM Python)**: JVM 안에서 Python을 직접 실행(프로세스 생성 없음). 성능 유리하나 지원 패키지 제약.
- **Jython**: JVM용 Python이지만 Python 2.7 수준이라 최신 문법/CPython 확장 미지원(사실상 비권장).
- **Py4J / gRPC / REST 마이크로서비스**: 장기 실행 파이썬 서버와 통신. 프로세스 재생성 비용이 없고 확장성이 좋지만 인프라가 복잡해진다.

각 방식은 "성능 · 배포 단순성 · 패키지 호환성" 사이의 트레이드오프가 다르다.
