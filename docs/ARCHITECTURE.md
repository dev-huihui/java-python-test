# 프로젝트 설명 — Java ↔ Python 검증 연동

이 문서는 이 프로젝트가 **무엇을·왜·어떻게** 하는지와,
지금 쓰고 있는 **두 가지 실행 방식(프론트/백엔드)의 장단점·주의점**,
그리고 **온라인 / 폐쇄망(오프라인) 환경에서의 동작 방식**을 정리한다.

> 앞으로 방식이 계속 추가될 수 있다. 새 내용을 넣을 때는 아래 구성에 맞춰
> — **실행 방식**은 §2, **선택 기준**은 §3, **환경별 동작**은 §5 — 같은 맥락끼리 모아 정리한다.

---

## 1. 핵심 아이디어

> **검증 규칙(로직)은 Python 파일에만 두고, Java 프로젝트는 그 Python 을 "실행하는 통로" 역할만 한다.**

- 검증 로직: `src/main/resources/python/` 의 `validation.py`(회원가입), `validators.py`(다양한 검증기)
- 실행 위치: **프론트엔드(브라우저)** 또는 **백엔드(서버)** 중 선택
- 같은 `.py` 파일을 두 위치에서 공용으로 사용하므로, 규칙을 바꾸면 한 곳(Python)만 수정하면 된다.

```
                ┌─────────────────────────────┐
                │   검증 규칙 (Python 파일)     │
                │  validation.py / validators.py │
                └───────────────┬─────────────┘
              (같은 파일을 두 경로가 공용)
          ┌───────────────────┴───────────────────┐
          ▼                                         ▼
 ① 프론트엔드 (브라우저)                    ② 백엔드 (서버, JVM)
   Pyodide(WASM)로 .py 실행                 ProcessBuilder로 python 실행
```

## 2. 실행 방식별 장단점·주의점

### 2.1 방식 ① — 프론트엔드 실행 (Pyodide)

**동작:** 브라우저가 [Pyodide](https://pyodide.org)(CPython을 WebAssembly로 포팅한 런타임)를 로드하고,
서버가 내려준 `.py` 텍스트(`/python/{name}`)를 그 자리에서 실행한다. 서버 왕복 없이 브라우저 안에서 검증이 끝난다.

**장점**
- 서버에 Python 설치가 필요 없다.
- 네트워크 왕복이 없어 입력 즉시 피드백(빠른 UX)이 가능하다.
- 서버 부하가 없다.

**단점 · 주의점**
- **보안 검증으로 쓰면 안 된다.** 브라우저에서 도는 코드는 사용자가 얼마든지 우회·조작할 수 있다. 어디까지나 *편의(UX)용 사전 검증*이며, **실제 신뢰가 필요한 검증은 반드시 서버에서 다시 해야 한다.**
- **검증 로직이 클라이언트에 그대로 노출**된다. `.py` 소스가 브라우저로 전송되므로, 민감한 규칙·비밀 값을 넣으면 안 된다.
- **초기 로딩이 무겁다.** Pyodide 런타임은 수 MB 이상이라 첫 실행 시 다운로드 지연이 있다. 로드 위치(CDN/로컬 번들)는 §5 에서 다룬다.
- Pyodide는 순수 Python + 일부 패키지만 지원한다. C 확장에 의존하는 라이브러리는 제약이 있을 수 있다(다수는 지원되지만 전부는 아님).
- 브라우저 탭에 Python 런타임이 상주하므로 메모리 사용이 늘어난다.

### 2.2 방식 ② — 백엔드 실행 (ProcessBuilder)

**동작:** 스프링 컨트롤러가 요청을 받으면 `PythonValidationService` 가 classpath 의 `.py` 를 임시 폴더로 꺼내고
`ProcessBuilder` 로 `python <script>` 를 실행한다. 입력은 **stdin(JSON)** 으로 넘기고 결과는 **stdout(JSON)** 으로 받는다.

**장점**
- 서버에서 실행되므로 **신뢰할 수 있는 검증**(위·변조 불가)이다.
- 검증 로직이 클라이언트에 노출되지 않는다.
- Java 에서 언어에 독립적으로 Python 자산을 재사용할 수 있다(연동 방식이 단순).

**단점 · 주의점**
- **매 요청마다 Python 프로세스를 새로 생성**한다. 프로세스 fork 비용(수십~수백 ms)이 있어 **고빈도 요청에는 비효율적**이다. 개선하려면 장기 실행 파이썬 프로세스/서버(Py4J·gRPC·REST 마이크로서비스), 배치 처리, 또는 JVM 내 실행(GraalPy) 등을 고려한다(§4).
- **서버 환경에 Python 설치·버전·경로 의존**이 생긴다. 배포 대상마다 Python 이 있어야 하고 `app.python.command` 를 맞춰야 한다.
- **인코딩 함정.** Windows 에서 파이프의 기본 인코딩이 시스템 로케일(예: CP949)이라 한글이 깨질 수 있다. 이 프로젝트는 `cli.py`/`field_cli.py` 에서 stdin/stdout 을 UTF-8 로 고정하고 `PYTHONIOENCODING=utf-8` 을 지정해 해결했다.
- **보안.** 사용자 입력은 stdin(JSON)으로만 전달하고 명령행 인자/셸로 넘기지 않아 명령어 주입을 피했다. 임의 스크립트/경로를 실행하지 않도록 실행 대상은 화이트리스트로 고정해야 한다.
- **동시성.** 요청당 1 프로세스라 동시 요청이 많으면 프로세스 수가 폭증할 수 있다. 실제 서비스에서는 실행 수 제한(세마포어/스레드풀)이 필요하다.
- **타임아웃·스트림 처리.** 무한 루프/행을 막기 위해 타임아웃(현재 10초)을 둔다. 또한 출력이 매우 클 경우 stdout/stderr 버퍼가 차면서 교착(deadlock)이 생길 수 있으니, 큰 출력이 예상되면 스트림을 별도 스레드로 읽거나 리다이렉트해야 한다(현재는 출력이 작아 문제없음).

### 2.3 한눈에 비교

| 항목 | 프론트(Pyodide) | 백엔드(ProcessBuilder) |
|------|-----------------|------------------------|
| 실행 위치 | 브라우저 | 서버(JVM) |
| 신뢰성(위·변조) | 낮음(우회 가능) | 높음 |
| 검증 로직 노출 | 노출됨 | 감춰짐 |
| 서버 Python 필요 | 불필요 | 필요 |
| 초기 비용 | Pyodide 로딩(수 MB) | 요청당 프로세스 생성 |
| 폐쇄망 | 로컬 번들이면 가능 | 가능 |

## 3. 언제 무엇을 쓰나

| 상황 | 권장 방식 |
|------|-----------|
| 입력 즉시 피드백 등 UX 편의 | 프론트(Pyodide) |
| 실제로 신뢰해야 하는 검증(저장·인증·결제 등) | **백엔드(서버)** |
| 오프라인/인터넷 불가 환경 | 백엔드 또는 Pyodide 로컬 번들 |
| 초당 많은 요청 처리 | 백엔드지만 프로세스 재사용/전용 서비스로 개선 |

**원칙:** 프론트 검증은 어디까지나 보조 수단이고, **최종 신뢰 경계는 서버**다. 두 방식을 함께 쓸 때도 서버 검증을 생략하면 안 된다.

## 4. 참고 — 다른 Java ↔ Python 연동 방식

이 프로젝트는 가장 단순한 **ProcessBuilder(외부 프로세스 호출)** 방식을 쓴다. 대안:

- **GraalPy (GraalVM Python)**: JVM 안에서 Python 을 직접 실행(프로세스 생성 없음). 성능 유리하나 지원 패키지 제약.
- **Jython**: JVM용 Python 이지만 Python 2.7 수준이라 최신 문법/CPython 확장 미지원(사실상 비권장).
- **Py4J / gRPC / REST 마이크로서비스**: 장기 실행 파이썬 서버와 통신. 프로세스 재생성 비용이 없고 확장성이 좋지만 인프라가 복잡해진다.

각 방식은 "성능 · 배포 단순성 · 패키지 호환성" 사이의 트레이드오프가 다르다.

## 5. 온라인 / 폐쇄망(오프라인) 환경에서의 동작

이 프로젝트의 **유일한 외부(인터넷) 의존**은 프론트엔드가 Pyodide 를 CDN(jsDelivr)에서 불러오는 부분이다. 나머지(백엔드 Python 실행, 앱 실행)는 모두 로컬이라 네트워크가 필요 없다.

### 5.1 구성별 요약

| 구성 | 온라인 | 폐쇄망(오프라인) | 비고 |
|------|:---:|:---:|------|
| 백엔드 검증 (ProcessBuilder → 로컬 Python) | ✅ | ✅ | 로컬 Python 실행, 네트워크 불필요 |
| 프론트 검증 · **온라인 로드**(CDN) | ✅ | ❌ | CDN 접근 불가 시 Pyodide 를 못 받음 |
| 프론트 검증 · **오프라인 로드**(로컬 번들) | ✅ | ✅ | `/pyodide/` 에서 로드 |
| Maven 빌드 | ✅ | ✅ * | `.m2` 캐시가 있으면 `mvnw -o` 로 가능 |
| 앱 실행 자체 | ✅ | ✅ | Spring Boot 서버는 네트워크 불필요 |

\* 최초 1회는 인터넷이 되는 곳에서 의존성을 받아 `.m2` 로컬 저장소에 캐시해 두어야 한다.

### 5.2 온라인 환경

- **프론트**: 온라인 페이지는 CDN 에서, 오프라인 페이지·선택은 로컬 번들에서 Pyodide 를 로드한다(둘 다 동작).
- **백엔드**: 로컬 Python 을 호출하므로 온라인 여부와 무관하다.

### 5.3 폐쇄망(오프라인) 환경

- **프론트 · 온라인 로드**: CDN 에 접근할 수 없어 **동작하지 않는다.**
- **프론트 · 오프라인 로드**: 프로젝트에 번들된 `src/main/resources/static/pyodide/` 에서 로드하므로 **정상 동작한다.**
- **백엔드**: 로컬 Python 실행이라 **정상 동작한다.**
- **빌드/배포**: 의존성이 `.m2` 에 캐시돼 있으면 `mvnw -o` 로 오프라인 빌드가 되고, 만들어진 WAR 에는 `static/pyodide/` 가 포함되므로 폐쇄망 배포 후에도 오프라인 프론트가 그대로 동작한다.

**정리:** 폐쇄망에서는 **백엔드 방식**과 **프론트의 오프라인 로드**만 사용하면 된다(온라인 로드 페이지는 피한다).

### 5.4 페이지 구성 (온라인/오프라인·프론트/백엔드)

| 메뉴 | 페이지 | Pyodide 로드 | 인터넷 |
|------|--------|--------------|:---:|
| 회원가입 · 프론트 온라인 | `frontend.html` | CDN(jsDelivr) | 필요 |
| 회원가입 · 프론트 오프라인 | `frontend-offline.html` | 로컬 번들 `/pyodide/` | 불필요 |
| 회원가입 · 백엔드 | `backend.html` | 없음 | 불필요 |
| 검증기 · 프론트 | `validators-frontend.html` | 화면 안 라디오로 온라인/오프라인 선택 | 선택에 따름 |
| 검증기 · 백엔드 | `validators-backend.html` | 없음(Pyodide 미사용) | 불필요 |

- 회원가입 프론트는 **온라인/오프라인이 각각 별도 페이지**다.
- 검증기는 **프론트/백엔드가 별도 페이지**이고, 프론트 페이지 안에서 **온라인/오프라인을 라디오로** 고른다. 로더(`pyodide.js`)는 head 에 고정하지 않고 검증을 처음 실행할 때 선택한 소스에서 **동적으로 주입**하며, 선택한 소스에 맞춰 `loadPyodide({ indexURL })` 를 호출한다(온라인=CDN URL, 오프라인=`/pyodide/`). 소스는 최초 1회 로드 시 고정되고, 바꾸려면 새로고침한다.
- 검증기 **백엔드 페이지는 Pyodide JS 를 전혀 내려받지 않는다.**

### 5.5 오프라인 번들 준비 방법

로컬 번들 파일은 `src/main/resources/static/pyodide/` 에 있다. 다시 받거나 버전을 올릴 때는 **인터넷이 되는 환경에서** 아래 스크립트를 실행한다.

```powershell
powershell -File scripts/fetch-pyodide.ps1
```

받는 파일은 `loadPyodide` 가 필요로 하는 코어(`pyodide.js`, `pyodide.mjs`, `pyodide.asm.js`, `pyodide.asm.wasm`, `python_stdlib.zip`, `pyodide-lock.json`)로 약 14MB 다. 검증 코드는 표준 라이브러리(`re`, `ipaddress`, `datetime`, `urllib`)만 사용하고 이는 Pyodide 코어에 기본 포함되므로 **런타임에 추가 패키지 다운로드가 없다.** 이 파일들만 있으면 프론트도 완전 오프라인으로 동작한다.
